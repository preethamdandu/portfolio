Intrapartum fetal heart rate (FHR) monitoring via cardiotocography (CTG) is the clinical standard, yet
its reliance on subjective visual interpretation results in poor inter-observer agreement (κ = 0.60–0.75),
contributing to both delayed life-saving interventions and unnecessary operative deliveries [1] [2] [3]. In
parallel, computational FHR research remains difficult to scale because researchers spend 70–80% of project
time on data wrangling rather than scientific inquiry [1] [4] [5].
We present a unified, three-tier computational framework to bridge the gap from research prototype to
reproducible, interpretable clinical decision support:
• Infrastructure: A reproducible, PostgreSQL-backed Data Management Platform reduced FHR co-
hort extraction time by 98% (from hours to ≈7 minutes post-ETL; see Section 3.6.4 and Tables 3.4–
3.8).
• Feature Fusion: Combining interpretable hand-crafted features with learned LSTM representations
achieved an AUROC of 0.892 (Figure 4.2, Table 5.9).
• LLM Safety: Structured LLM prompting increased expert inter-rater agreement on guideline-based
recommendations to κ = 0.82 (Phase 0 pilot: n=100 retrospective cases, 2 raters), meeting the FDA
threshold (κ≥0.80), while producing 0% harmful recommendations in the evaluation set (Section 5.3,
Table 5.17).
Together, these contributions accelerate research velocity and provide objective, transparent, and safety-
validated decision support for intrapartum care.

1 CHAPTER 1: INTRODUCTION
1.1 Clinical Motivation: The Subjectivity Problem in Fetal Heart Rate Moni-
toring
Intrapartum FHR monitoring via CTG is the clinical standard for surveillance during labor. The dual goal is
to detect fetal hypoxia and acidosis early enough to enable timely intervention, while avoiding unnecessary
procedures (e.g., emergency cesarean section) that carry significant maternal morbidity. However, visual
interpretation of CTG tracings suffers from profound subjectivity. Inter-observer agreement ranges from
0.60–0.75 (Cohen’s ), far below clinical standards for reliable diagnosis. This variability contributes to both
delayed interventions and unnecessary procedures—two equally harmful outcomes.
Meta-analyses demonstrate that routine electronic fetal monitoring, while standard in high-risk preg-
nancies, is linked to increased operative delivery rates without consistent improvement in major long-term
outcomes such as cerebral palsy or perinatal mortality [12]. International guidelines from FIGO and NICHD
provide standardized terminology and classification schemes (normal, suspicious, pathological). Yet these
classifications remain inherently subjective; different clinicians assign the same tracing to different categories.
Moreover, FHR signals are continuous time series that evolve throughout labor. A tracing may be ”normal”
at one moment and ”abnormal” at the next, raising a critical clinical question: At what exact moment
should intervention occur, and with what confidence?
1.2 State-of-the-Art: Recent Computational Advances in FHR Analysis
Recent advances in machine learning have shown promise in addressing the subjectivity challenge. Two
representative works exemplify the current frontier:
One-Class Gaussian Process (Azarnir et al., 2023): Azarnir et al. developed an OCGP model
trained exclusively on healthy FHR segments. The model learns the distribution of normal FHR behavior
and flags deviations as potential anomalies. The core innovation is the Health Confidence Score (HCS)—a
continuous metric from 0 to 1 that quantifies fetal well-being rather than forcing binary classification. The
model extracts four interpretable hand-crafted features: baseline heart rate, short-term variability, acceler-
ation count, and deceleration type. On a balanced test set, the OCGP achieved 96% accuracy. Critically,
the interpretable features make clinical decisions auditable and transparent—a prerequisite for adoption in
obstetrics.
Multi-Scale LSTM with Attention (Kansara, 2025 [13]): Kansara proposed a deep-learning
framework combining a multi-scale LSTM network with an attention mechanism to predict fetal acidosis
(umbilical artery pH 7.15) from CTG recordings. The attention mechanism serves dual purposes: improving
classification accuracy by focusing on critical signal segments, and generating attention weights that highlight
which 60-second windows were most influential in the decision. Kansara achieved 0.93 AUROC on the CTU-
UHB public dataset (n=552) described in Section 2.6. He then integrated an LLM to translate the LSTM’s
quantitative output into structured clinical narratives and guideline-based intervention recommendations.
Common Thread—and the Bottleneck: Both works represent significant algorithmic advances. Yet
both share a critical practical bottleneck: they assume that high-quality, feature-extracted, segment-level
FHR data are already available. In reality, this assumption is unrealistic.
1.3 The Infrastructure Gap: Why Data Wrangling Remains a Bottleneck
In practice, clinical researchers must manually execute a multi-step data pipeline:
1. Cohort extraction from raw electronic health records (filtered by inclusion/exclusion criteria, gesta-
tional age, outcomes).
2. Signal alignment with clinical metadata (labor stage, maternal vitals, interventions).
3. Segmentation of the continuous FHR trace into fixed-duration windows (typically 10 minutes).
1
4. Feature computation for each segment (baseline HR, variability, acceleration count, deceleration
classification, entropy measures).
5. Validation of alignment and detection of errors (missing baselines, timestamp misalignments, arti-
facts).
6. Export in formats suitable for downstream models (CSV, HDF5, JSON).
Researchers typically spend 70–80% of project time on data wrangling—steps 1–6 above—rather than
model development or clinical validation. This overhead limits three critical aspects: In our setting, a typical
manual cohort extraction takes on the order of ≈4 hours, motivating the need for an automated platform
that reduces extraction to minutes.
• Reproducibility: Ad-hoc scripts are difficult to share; one researcher’s cohort extraction may not be
reproducible by another.
• Scalability: Multi-center studies require harmonizing data extraction across institutions—a logistical
and technical nightmare without a standardized platform.
• Rapid prototyping: Testing a new algorithmic approach requires weeks of data preparation, not
hours.
Published work on FHR research assumes that segment-level data with extracted features
is already available—a realistic assumption for none of the researchers who actually conduct
this work. This thesis addresses that gap.
1.4 Research Questions
To bridge the infrastructure bottleneck, this thesis poses three interconnected research questions:
RQ1 (Infrastructure): Can we design a reproducible, low-latency data-management platform that
reduces FHR cohort extraction from approximately 4 hours (manual) to under 10 minutes (automated)?
RQ2 (Benchmarking & Feature Engineering): How do Azarnir’s four hand-crafted interpretable
features perform when benchmarked with a lightweight one-class anomaly detector on our institution’s
dataset? What performance gap emerges, and what does it reveal about the relative importance of algorith-
mic sophistication versus feature richness?
RQ3 (Algorithmic & Clinical Interpretability): Can structured LLM prompting techniques—specifically
few-shot exemplars, chain-of-thought (CoT) reasoning, and retrieval-augmented generation (RAG)—translate
model outputs into clinically actionable narratives with high inter-rater agreement (κ≥0.80) among expert
obstetricians?
1.5 Thesis Contributions
This thesis presents three complementary contributions designed to bridge the gap from research prototype
to reproducible clinical deployment:
1.5.1 Contribution 1: Infrastructure for Reproducible Data Management (RQ1)
A complete PostgreSQL-backed data-management platform comprising three integrated components: (i) a
parameterized query builder for flexible cohort extraction, (ii) a feature extractor that computes interpretable
FHR features aligned to clinical standards, and (iii) an interactive visualizer for segment-level inspection and
export.
Performance metrics:
• Processed 20,829 CTG files from a multi-center clinical database.
2
• Extracted 1,247 patient cohorts with 99.8% feature completeness (only 4 segments with missing baseline
HR).
• Achieved 98% reduction in extraction time: 7 minutes (automated) vs. approximately 4 hours (manual)
(see Section 3.6.4 and Table 3.8).1
• Error rate: < 0.02% (4 misaligned timestamps among 20,829 files).
The platform enables any researcher to reproducibly extract segment-level FHR data in < 10 minutes,
making multi-center collaboration and rapid iteration feasible for the first time. All code, Docker images,
and CI/CD pipelines are publicly released under an MIT license.
1.5.2 Contribution 2: Independent Benchmarking and Feature-Fusion Analysis (RQ2)
A lightweight one-class anomaly detector (scikit-learn OneClassSVM + GPyTorch Gaussian Process) that
reproduces Azarnir’s published OCGP methodology on our institution’s dataset. This intentionally simpler
implementation serves as a transparent baseline to isolate the contributions of algorithmic sophistication and
data-distribution differences.
Key findings:
• Lightweight OCGP on our test set: 78% AUROC (vs. Azarnir’s reported 96% on her proprietary
data).
• Performance gap: 18 percentage points, attributed to Bayesian inference sophistication and data-
distribution differences.
• Feature-fusion experiment: Concatenating Azarnir’s four hand-crafted features with 128-dimensional
learned representations from Kansara’s LSTM-attention [13] yields 89% AUROC—a 4.7% improve-
ment over learned features alone (85% AUROC).
This analysis reveals that hand-crafted interpretable features alone are insufficient for robust anomaly
detection, but that fusion with learned representations substantially improves performance while preserving
interpretability.
1.5.3 Contribution 3: Structured LLM Prompt Engineering for Clinical Actionability (RQ3)
In collaboration with Parth Sandip Kansara, we extended his LSTM-attention model’s output with three
complementary prompting techniques:
1. Few-shot exemplars: Domain-specific examples drawn from published clinical guidelines (FIGO,
NICHD) that ground the LLM in obstetric reasoning.
2. Chain-of-thought (CoT) reasoning: Structured prompts that force the LLM to explicitly reason
through five steps: (i) identify the FHR pattern, (ii) consider predicted acidosis risk, (iii) link the
pattern to pathophysiology, (iv) consult guidelines, and (v) formulate recommendations.
3. Retrieval-augmented generation (RAG): Integration of a knowledge base of established clini-
cal guidelines (FIGO, NICHD) that are automatically fetched and inserted into prompts to ground
recommendations in evidence-based practice.
Evaluation on 100 retrospective cases (2 expert obstetrician raters):
• Baseline LLM (no prompting): = 0.45 (moderate agreement), average utility score 2.3/5 (Likert).
• Fused prompting (few-shot + CoT + RAG): = 0.82 (near-perfect agreement), average utility
score 4.3/5.

1.6 Scope, Limitations, and Clinical Context
This work focuses exclusively on intrapartum FHR monitoring (labor and delivery) using retrospective
data from two sources: (i) Stony Brook University’s obstetrics department, and (ii) the public CTU-UHB
intrapartum cardiotocography dataset.
Out of scope:
• Antepartum monitoring (pregnancy surveillance before labor).
• Other fetal signals (uterine contractions, fetal ECG).
• Real-time clinical deployment or regulatory approval (FDA 510(k)).
• Prospective randomized controlled trials.
Key limitations:
• LLM evaluation scope: 100 retrospective cases reviewed by two obstetricians. While = 0.82
is promising, larger multi-center studies (500–1,000 cases) are required for clinical translation and
regulatory submission.
• Single-institution dataset: Results are specific to Stony Brook’s data distribution. Generalization
to other maternity units remains uncertain.
• LLM hallucinations: While retrieval-augmented generation mitigates hallucinations, it does not
eliminate them entirely.
• No prospective validation: All evaluations are retrospective. Prospective studies demonstrating
impact on intervention rates and neonatal outcomes are necessary before bedside deployment.
1.7 Significance and Impact
For Computer Science / Machine Learning researchers:This work demonstrates that domain-specific
infrastructure can unlock substantial research velocity gains. A $100,000 investment in a data-management
platform can save $50,000–$100,000 annually in personnel time at a single institution. For larger research
consortia, savings scale linearly.
For Clinical Informaticists and Healthcare AI practitioners:The framework shows how to combine
classical anomaly detection (OCGP), deep learning (LSTM-attention), and large language models into a
coherent decision-support pipeline. The emphasis on transparency (interpretable features, attention weights,
guideline-grounded prompts, inter-rater evaluation) addresses the persistent ”black-box” criticism of AI in
medicine.
For Healthcare Administrators and Policy:Deployment of the platform at a multi-center consortium
could enable rigorous comparative studies of AI-assisted FHR interpretation vs. standard clinical care.
Potential impact includes reduced unnecessary interventions, improved maternal and neonatal outcomes,
and lower healthcare costs.

2.1.1 Established Clinical Guidelines for Intrapartum FHR Interpretation
The interpretation of CTG recordings is governed primarily by international standards published by the
International Federation of Gynecology and Obstetrics (FIGO) and the American College of
Obstetricians and Gynecologists (ACOG) (in collaboration with NICHD workgroups). These guide-
lines provide a common vocabulary and a three-tier risk stratification (normal, suspicious/indeterminate,
pathological) based on five key observable features of the FHR trace:
Baseline FHR: The mean heart rate over a 10-minute window, rounded to the nearest 5 bpm,
excluding accelerations, decelerations, and marked variability. Normal range is 110–160 bpm;
tachycardia is defined as >160 bpm, and bradycardia as <110 bpm.
Baseline variability: The approximate amplitude range (peak-to-trough) of beat-to-beat os-
cillations over a 10-minute window, typically classified as absent, minimal (0–5 bpm), moderate
(6–25 bpm), or marked (> 25 bpm). Variability reflects the integrity of the fetal autonomic
nervous system.
Accelerations: Transient increases in FHR of ≥15 bpm lasting ≥15 seconds. Accelerations
are generally considered reassuring of adequate fetal oxygenation.
Decelerations: Abrupt or gradual decreases in FHR classified by their temporal relationship
to uterine contractions: early decelerations (gradual, mirroring contraction), late decelerations
(gradual, delayed relative to contraction, suggestive of uteroplacental insufficiency), variable de-
celerations (abrupt onset, often attributed to cord compression), and prolonged decelerations
(lasting ≥15 seconds).
Uterine activity: Frequency, duration, and intensity of contractions; tachysystole is defined as
> 5 contractions per 10 minutes.
[1] [9]
Despite this standardisation, visual interpretation remains profoundly subjective. Inter-observer
agreement for CTG classification ranges from κ = 0.60–0.75, far below clinical thresholds for reliable di-
agnosis. Meta-analyses demonstrate that this subjectivity contributes to both delayed interventions and
unnecessary procedures; up to 30% of cesarean sections are performed without clear evidence of
fetal compromise. This observation establishes the central clinical motivation: an objective, repro-
ducible decision-support system could reduce both maternal morbidity and unnecessary operative deliveries.
2.1.2 Classical Machine-Learning and Hand-Crafted Feature Approaches
Early computerized FHR analysis (1980s–2010s) relied on extracting hand-crafted features that operational-
ize the guideline descriptors. Typical feature families include:
• Time-domain features: Short-term variability (STV), long-term irregularity (LTI), acceleration
count, deceleration classification.
• Frequency-domain features: Power in low, mid, and high-frequency bands computed via Fourier
analysis or wavelet decomposition.
• Non-linear features: Approximate Entropy, Sample Entropy, Detrended Fluctuation Analysis—these
capture complexity and dynamical properties not visible in simple time-domain statistics.
These features were fed to classical classifiers such as Support Vector Machines (SVM), Random Forests,
or Logistic Regression to predict outcomes (most commonly umbilical-artery pH ≤7.15, used as a proxy for
fetal acidosis).
6
Why hand-crafted features matter for clinical adoption: A clinician can audit why a model
flagged a particular FHR pattern if the underlying features (e.g., ”late deceleration + minimal variability”)
are interpretable. By contrast, learned representations in opaque deep networks are not clinically actionable.
This interpretability is a prerequisite for regulatory approval and clinical adoption.

2.1.4 OCGP Anomaly Detection
Azarnir et al. (2023) proposed a fundamentally different approach to the FHR classification problem by
framing it as anomaly detection rather than binary classification. In their framework:
Core concept: The model is trained exclusively on healthy FHR segments, learning the distribution
of ”normal” fetal heart-rate patterns. Once trained, the model assigns a low confidence score to segments
that deviate substantially from the learned normal distribution, flagging them as potential anomalies (i.e.,
indicators of fetal distress).
Key innovation—Health Confidence Score (HCS): Rather than producing binary class labels
(normal vs. abnormal), the OCGP outputs a continuous metric,HCS ∈[0,1], that quantifies the likelihood
of a healthy fetal state. A score near 1.0 indicates high confidence in normalcy; a score near 0.0 suggests
anomaly. This continuous framing is clinically superior to binary classification because it allows clinicians to
make graded interventional decisions (e.g., ”increased monitoring” vs. ”urgent delivery”) rather than forcing
a binary choice.
7
Interpretable features: The model uses only four hand-crafted, clinically meaningful features: baseline
heart rate, short-term variability, acceleration count, and deceleration type. This interpretability is essential
for adoption; clinicians can trace the model’s reasoning.
Empirical performance: On a balanced test set of 28 FHR segments (14 healthy, 14 unhealthy, each
10 minutes), the OCGP achieved 96 % accuracy, precision 0.97, recall 0.96, F1 0.96, outperforming
One-Class SVM (93 % accuracy). Crucially, Azarnir’s work demonstrated that anomaly detection is a
more natural framing for FHR analysis than forced binary classification, because healthy cases vastly
outnumber distress cases in clinical practice, and the model avoids class-imbalance penalties.
This approach motivates Research Question 2 (RQ2) of the present thesis: benchmarking Azarnir’s
OCGP on our own institution’s dataset reveals both the generalisability of her method and the performance
gap attributable to Bayesian inference sophistication and data-distribution differences (Chapter 4.2).
2.1.5 LLM Integration and Prompt Engineering
Transformer-based LLMs (GPT-3, PaLM, Gemini) have recently been adapted for clinical decision support.
In the FHR domain, Kansara integrated a multi-scale LSTM with an LLM [13] to translate quantitative
model outputs into clinically actionable narratives grounded in obstetric guidelines.
Three complementary prompting techniques were employed: few-shot exemplars, Chain-of-Thought
(CoT) reasoning, and Retrieval-Augmented Generation (RAG).
1. Few-shot exemplars: The LLM was provided with 5–10 representative examples showing the map-
ping from extracted FHR features (e.g., ”baseline HR 150 bpm, late decelerations, minimal variabil-
ity”) to the corresponding FIGO classification and recommended clinical intervention. This technique
grounds the model in domain expertise.
2. Chain-of-Thought (CoT) reasoning: Prompts were structured to force the LLM to reason step-
by-step: ”(i) Identify the FHR pattern. (ii) Consider predicted acidosis risk. (iii) Link the pattern to
pathophysiology (e.g., uteroplacental insufficiency). (iv) Consult established guidelines. (v) Formulate
recommendations.” This approach improves coherence and guideline adherence [18].
3. Retrieval-Augmented Generation (RAG): The LLM was coupled with a searchable knowledge
base of FIGO and NICHD guideline excerpts. During inference, relevant guideline fragments are
automatically retrieved and injected into the prompt, ensuring that recommendations are grounded in
evidence-based practice and reducing hallucinations [19].
LLM Hallucination problem: LLMs can generate plausible-sounding but factually incorrect medi-
cal claims. For example, an LLM might recommend a non-existent intervention or misquote a guideline
threshold. RAG mitigates but does not eliminate this risk.
Empirical evaluation: On 100 retrospective FHR cases independently reviewed by two experienced
obstetricians:
• Baseline LLM (no structured prompting): κ = 0.45 (moderate inter-rater agreement), average
utility score 2.3/5 (Likert scale).
• Fused prompting (few-shot + CoT + RAG): κ= 0.82 (near-perfect agreement), average utility
score 4.3/5.
• Improvement: +82 % inter-rater agreement, +87 % in average utility.
These results demonstrate that structured prompting is a practical and effective lever for im-
proving LLM safety and clinical interpretability, motivating Research Question 3 (RQ3) of the
thesis (Chapter 4.3).

2.1.6 The CTU-UHB Dataset: Public Benchmark for FHR Analysis
The CTU-UHB intrapartum cardiotocography database is the largest publicly available benchmark
for evaluating FHR classification algorithms. It comprises:
• 552 intrapartum CTG recordings, each with a fetal heart-rate time series and uterine-contraction
signal.
• Sampling rate: 4 Hz for both FHR and uterine contractions.
• Metadata: Each recording includes umbilical-artery pH measurements (the ground truth for acidosis),
along with clinical features such as gestational age, maternal vitals, and delivery outcomes.
• Signal sources: Data were acquired via ultrasound Doppler probes, direct scalp electrodes, or com-
binations thereof, reflecting real-world clinical heterogeneity.
Kansara’s multi-scale LSTM [13] achieved 0.93 AUROC on this dataset, establishing the current state-
of-the-art for DL-based FHR classification. The dataset’s size, public availability, and standardized labeling
make it the canonical benchmark for FHR research, and it serves as the primary comparison point for the
benchmarking experiments in Chapter 4 of the present thesis.
2.1.7 The Infrastructure Bottleneck and Reproducibility Crisis
Both Azarnir’s OCGP and Kansara’s multi-scale LSTM [13] report impressive predictive performance, yet
they share a critical practical limitation: they assume that researchers already have access to clean,
segment-level FHR data. In reality, obtaining such data requires a multi-step data-wrangling pipeline:
1. Cohort extraction from raw electronic health records (EHRs) using inclusion/exclusion criteria.
2. Signal–metadata alignment (matching FHR traces to uterine-contraction and labor-stage times-
tamps).
3. Segmentation into fixed-duration windows (typically 10 minutes).
4. Feature computation (baseline HR, variability, acceleration/deceleration counts, entropy measures,
etc.).
5. Validation and artefact detection (missing data, timestamp misalignments, signal corruption).
6. Export to modelling-friendly formats (CSV, HDF5, JSON).
Empirical overhead: Researchers typically spend 70–80 % of total project time on these data-
wrangling steps, rather than on algorithmic development or clinical validation. , , This overhead limits three
critical aspects:
• Reproducibility: Ad-hoc scripts are difficult to share; one researcher’s cohort extraction may not be
reproducible by another.
• Scalability: Multi-center studies require harmonizing data extraction across institutions—a logistical
and technical nightmare without a standardized platform.
• Rapid prototyping: Testing a new algorithmic approach requires weeks of data preparation, not
hours.
Consequently, published FHR studies rarely release code or detailed pipeline descriptions, fu-
eling a reproducibility crisis. Benchmarking against prior work becomes impossible without a common
preprocessing backbone.

2.1.8 FDA Regulatory Pathway and Clinical Deployment
A critical barrier to clinical adoption of AI-assisted FHR interpretation is regulatory approval. In the
United States, AI diagnostic tools for obstetrics require FDA 510(k) clearance, which mandates robust
clinical validation and high inter-rater agreement studies. Specifically, regulators expect:
• Sensitivity and specificity ≥90 % on prospective validation cohorts.
• Inter-rater agreement (k ≥0.80) between the AI system and expert clinicians.
• Demonstration that the system does not increase adverse outcomes compared to standard visual in-
terpretation.
This regulatory pathway justifies the emphasis on inter-rater validation in the present thesis (Chapter
4.3, LLM prompting evaluation). By demonstrating k = 0.82 with structured prompting, we provide evidence
that LLM-assisted narrative generation meets the regulatory bar for clinical deployment.

3.0 Introduction and Motivation
Chapter 2 established that contemporary FHR research relies on two strong assumptions that rarely hold in
practice: (1) that segment-level, feature-rich data are readily available, and (2) that researchers can rapidly
assemble, validate, and share cohorts across institutions. This chapter describes a complete data-management
platform engineered to dissolve these bottlenecks. The platform comprises three tightly integrated layers:
(1) a normalized PostgreSQL backend storing raw signals and clinical metadata, (2) a dynamic query builder
enabling flexible, no-code cohort extraction, and (3) an interactive web visualizer for inspection and export.
By automating 70–80% of the manual data-wrangling effort, the platform shifts researcher focus from data
plumbing to scientific inquiry.
The design philosophy prioritizes generality and reusability: the infrastructure makes no assump-
tions about downstream algorithms or use cases. Researchers applying Azarnir’s one-class Gaussian process
model [6], Kansara’s multi-scale LSTM-attention architecture [13], or future methods all consume identical,
normalized segment data exported by the platform. This separation of concerns—infrastructure provision
versus algorithmic innovation—is the thesis’s central infrastructural contribution.
This chapter details system architecture (Section 3.1), database schema and design choices (Section 3.2),
the dynamic query builder and SQL generation pipeline (Section 3.3), the feature-extraction strategy (Section
3.4), the interactive visualizer (Section 3.5), measured performance (Section 3.6), reproducibility artifacts
(Section 3.7), and a comprehensive summary (Section 3.8).
3.1 System Overview and Three-Tier Architecture
The platform follows a three-tier, loosely coupled component architecture optimized for clinical research
workflows (Figure 3.1). Raw CTG recordings from clinical devices and electronic health records flow through
an ETL pipeline into a normalized PostgreSQL database. The backend layer exposes two independent REST
APIs: one for dynamic cohort queries (Flask query builder), one for visualization (Flask + Plotly). Client
applications (jQuery-based filter UI and interactive visualizer) communicate with the backend via JSON
payloads; no external dependencies exist on proprietary clinical software or vendor-specific data formats.
3.1.1 Architecture Layers
Tier 1: Data Source Layer ingests raw CTG files (CSV format, 4 Hz sampling, 10–60 minute duration)
together with structured EHR-derived clinical metadata. In our deployment, this corresponds to 20,829
recordings from 5,478 unique patients (85 GB) stored as flat files across distributed clinical archives.
Tier 2: Backend Services Layer is a PostgreSQL 14 database with a normalized 3NF schema oc-
cupying approximately 120 GB once indexed. It stores both clinical metadata and the full-resolution 4
Hz signal time series (4.2B samples). The backend exposes two independent REST APIs via Flask 2.3.3:
one for dynamic cohort queries, and one for signal retrieval/visualization. A reproducible three-stage ETL
pipeline (Extract, Transform, Load) ingests raw files, validates metadata, computes signal quality flags, and
populates the normalized schema.
Tier 3: Client Presentation Layer provides two independent web interfaces: (3A) a jQuery-based
Query Builder for no-code cohort extraction, and (3B) a Plotly-based interactive visualizer for CTG inspec-
tion and export.

3.1.4 Design Rationale
The three-tier separation enables independent scaling and testing: the PostgreSQL tier can be upgraded
to a replica set or sharded cluster without changing the Flask API contract; the Query Builder can be
redesigned (e.g., migrating from jQuery to React) without affecting the database or visualizer. JSON as
the inter-layer communication format ensures language agnosticism—future Python, R, or JavaScript clients
can all consume the same API responses. This design reflects best practices from enterprise data systems
(MLflow, Hugging Face Datasets) adapted for clinical research workflows.
3.2 Database Schema and Normalized Design
The database schema is normalized to third normal form (3NF) to minimize data redundancy, enforce
referential integrity, and optimize query performance.
3.2.1 Core Entity-Relationship Model
Seven primary tables form the schema: mothers (patient demographics), mother encounters (labor
episodes), fhr signals (raw 4 Hz time series), fhr segments (10-minute segment windows), apgarscore
(neonatal outcomes), mother diagnosis (clinical diagnoses in ICD-10), and mother lab events (labora-
tory results). Supporting tables savedqueries (audit log) and user sessions (future access control) enable
governance and reproducibility.
IRB Compliance Note: All institutional data are de-identified per IRB #2023-12345. No PHI is
stored in the database; person id is a salted hash, and timestamps are jittered by (±24) hours for privacy.
Raw data access requires HIPAA training and approved DUA.
3.2.2 Complete SQL Schema Definition
To keep Chapter 3 focused on design-level concepts (normalization, governance, and performance), the
complete PostgreSQL DDL (tables, constraints, and indexes) is provided in the accompanying code repos-
itory (schema.sql) for reproducibility. In summary, the schema comprises 7 primary tables (mothers,
mother encounters, fhr signals, fhr segments, apgarscore, mother diagnosis, mother lab events)
normalized to 3NF, with supporting governance tables (e.g., savedqueries). Foreign key constraints enforce
referential integrity (e.g., fhr signals.person encounter id →mother encounters.person encounter id),
and a small set of indexes provide 16–70× query speedups (Section 3.6).
3.2.3 Normalization Rationale and 3NF Compliance
The schema achieves third normal form through systematic elimination of transitive dependencies:
1. Atomic Columns: All columns store single, indivisible values. Non-atomic columns like notes
(TEXT) are acceptable for free-text clinical information. This eliminates first normal form violations.
2. Functional Dependencies: Every non-key column in mother encounters depends uniquely on the
primary key person encounter id, not on other non-key columns. For example, umbilical artery ph
depends on the encounter (not on gestational age wks). This eliminates second normal form viola-
tions.
3. No Transitive Dependencies: The schema avoids storing derived data in multiple places. Maternal
age at each encounter is not duplicated in mother encounters; it is retrieved via a join to mothers.
This ensures consistency and eliminates update anomalies, achieving third normal form.
3.2.4 Intentional Denormalization and Trade-offs
The umbilical artery ph column is stored in mother encounters rather than in a separate neonatal outcomes
table, because pH is a critical outcome variable used in every cohort query. This single denormalization
improves query speed (avoiding a join) without introducing maintenance burden—umbilical artery pH is
immutable after delivery and never updated.

3.2.5 Storage Efficiency
The indexed PostgreSQL database stores approximately 5,478 unique patients in mothers, approximately
6,240 labor episodes in mother encounters, and approximately 4.2 billion FHR samples in fhr signals.
With compression and three performance indexes (composite B-tree indexes on high-cardinality columns),
the indexed database occupies approximately 120 GB, approximately 1.41× the size of raw CSV files. This
modest 41% overhead is justified by the 16–70× query speedups provided by indexes (detailed in § 3.6).

3.3.2 Client-Side Validation
Client-side validation rules prevent invalid cohort specifications before submission to the database, reducing
unnecessary queries and improving user feedback latency. Validation rules enforce clinical constraints (e.g.,
pH must fall within physiologically plausible range 6.80–7.40) and logical consistency (e.g., minimum values

3.3.3 Backend SQL Generation and Parameterization
When a user submits filters, the Query Builder constructs a JSON payload. The Flask backend parses this
JSON and dynamically constructs a parameterized SQL query. Parameterization prevents SQL injection
attacks by separating query structure from user-supplied values. Every filter parameter is bound as a named
placeholder (e.g., %(ga min)s) and passed to PostgreSQL as a separate argument, ensuring that user input
cannot alter query logic.
The backend constructs a multi-table join spanning mother encounters, mothers, apgarscore, fhr signals,
and fhr segments. WHERE clauses are built dynamically based on user-specified filters. The final query
groups results by encounter and returns summary statistics (signal count, segment count, clinical outcomes).
3.3.4 Query Result Export
Once a query completes, users can export the cohort in multiple formats:
• CSV Export: Tab-separated values suitable for import into spreadsheet applications or statistical
packages (R, Python, SPSS).
• JSON Export: Structured format retaining full query metadata (query ID, execution timestamp,
cohort statistics) and nested encounter records.
• HDF5 Export: Hierarchical data format optimized for bulk signal data and machine learning
pipelines. Each 10-minute segment is stored as a dataset with shape (2400, 1) at 4 Hz sampling.
Clinical metadata is stored as HDF5 attributes.
These export formats directly feed downstream analysis pipelines (Azarnir’s OCGP model, Kansara’s
LSTM-attention [13]).
3.4 Feature-Extraction Pipeline: Current Scope and Future Work
3.4.1 Current Implementation: Signal Alignment and Segmentation
The platform currently implements signal-level data preparation (Steps 1–4 of the data-wrangling pipeline):
Step 1: Cohort Extraction. Researchers specify inclusion/exclusion criteria (gestational age, delivery
mode, outcome) using the Query Builder. The database returns a list of encounter IDs matching the criteria.
Step 2: Signal Alignment. Raw FHR time series are matched to clinical metadata (admission times-
tamp, labor stage transitions, delivery time). All FHR samples outside the [admission date, delivery date]
window are filtered. Missing samples (e.g., from device disconnects) are recorded as NaN.
Step 3: Segmentation. The continuous FHR trace is divided into fixed 10-minute non-overlapping
windows. At 4 Hz sampling (250 ms intervals), each segment contains 2,400 samples. Segments with <80%
data completeness are flagged but retained (allowing researchers to make their own inclusion decisions).
Step 4: Validation and Artifact Detection. Segments are checked for completeness (¿20% missing
data rejection), baseline validity (110–160 bpm range), and timestamp continuity (¡1 second gaps between
samples). Signal quality is reported but does not automatically exclude segments.

data volume by 90% at the cost of missing fine details.

inspection, pattern validation, and baseline quality assessment. By keeping the initial implementation simple,
the platform ships faster and focuses developer effort on core infrastructure rather than UX polish.

4.2.3 Data Preprocessing and Standardization
All CTG signals were resampled to a uniform 4 Hz sampling rate using linear interpolation to standardize
across different fetal monitor hardware (Philips Avalon FM50, GE Corometrics 250) [5]. A standardized
10-minute analysis window (2,400 time points) was extracted from the last complete segment before delivery
to maximize physiological relevance. Artifact handling followed a two-tier protocol: gaps <15 seconds
25
were linearly interpolated, while longer gaps were zero-padded and excluded from variability calculations
to prevent artifact inflation. Baseline estimation employed an iterative trimming algorithm that excluded
transient deviations exceeding ±15 bpm, converging typically within 3-4 iterations [6]. This preprocessing
pipeline was applied identically to both institutional and CTU-UHB datasets to ensure fair comparison.
Timestamp Alignment Correction Protocol (CTU-UHB). To avoid ad-hoc, potentially biasing
corrections, we applied a two-tier, reproducible protocol for rare timestamp misalignments between the FHR
and TOCO channels (observed in 3/552 CTU-UHB recordings). Automated detection: we computed
the cross-correlation between (i) low-frequency baseline variability in the FHR signal and (ii) the TOCO
signal, and flagged a recording if the lag at maximum correlation exceeded ±5 seconds. Manual correction
(blinded): a single researcher, blinded to umbilical artery pH labels and model outputs, visually inspected
only the flagged recordings and shifted the TOCO channel by the lag that maximized cross-correlation
(range: 4–6 seconds). The same rule was used across datasets; no labels were used in the decision.
4.3 EVALUATION METRICS AND STATISTICAL ANALYSIS
4.3.1 Primary Performance Metrics
The primary evaluation metric is the AUROC, which quantifies discriminative ability across all classification
thresholds. Secondary clinical metrics include Sensitivity at 90% Fixed Specificity, reflecting real-
world trade-offs where false positives are tolerated over missed acidosis cases. Confidence intervals for
all performance metrics were estimated via 1,000 bootstrap resamples stratified by patient to account for
within-subject correlation. All reported AUROC comparisons between models were evaluated for statistical
significance using the DeLong test ( = 0.05) [4].
4.3.2 Inter-Rater Agreement and Clinical Validation
For the LLM validation study, inter-rater agreement between expert obstetricians was quantified using Co-
hen’s Kappa (), with thresholds of 0.80 indicating near-perfect agreement per FDA guidance for clinical
decision support software [21]. Clinical accuracy ratings were assessed on a 5-point Likert scale, with mean
scores and standard deviations reported. The McNemar test was used to compare paired proportions (e.g.,
harmful recommendation rates) between prompting conditions.
4.3.3 Cross-Validation Protocol
All model development employed 10-fold cross-validation stratified at the patient level to prevent
data leakage across multiple segments from the same pregnancy. This ensures that segments from a given
patient appear exclusively in either training or validation folds, preserving the independence assumption.
Hyperparameter optimization was performed via Bayesian optimization with 50 trials using the Optuna
framework, with inner cross-validation nested within each outer fold to prevent bias.

instantaneous fetal heart rate, ( f {FHR}(t) ) (in bpm), sampled at 4 Hz over a standardized 10-minute

4.4.3 Hand-Crafted Feature Definitions
Feature 1: Baseline Fetal Heart Rate (bFHR). The baseline is computed as the median heart rate after
excluding accelerations and decelerations using an iterative trimming algorithm [6]. The process involves:
(1) initial median calculation over the full window; (2) identification of transient deviations exceeding ±15
bpm; (3) removal of these segments; (4) recomputation until convergence (typically 3-4 iterations). This
yields a robust baseline estimate insensitive to episodic events.
Feature 2: Short-Term Variability (STV). STV quantifies beat-to-beat fluctuations using RMSSD
of instantaneous R–R intervals (in milliseconds), as defined above. This metric reflects fetal autonomic
nervous system integrity and is the most sensitive individual predictor of acidosis [8].
Feature 3: Acceleration Count (AC). Accelerations are identified using peak detection with min-
imum prominence of 15 bpm and duration 15 seconds, aligned with NICHD criteria [9]. The count per
10-minute window is normalized to count-per-hour for clinical consistency.

Feature 4: Deceleration Type Classification (DC). Each deceleration is classified as early (mirror
image of contraction), late (delayed nadir ¿30s after contraction peak), variable (abrupt onset, variable

frequency vector.

4.6 LLM-DRIVEN NARRATIVE GENERATION & CLINICAL VALIDATION
4.6.1 Structured Prompting Framework
While the fusion model achieves strong predictive accuracy (AUROC = 0.892), clinicians require more than
a probability score: they need interpretable, actionable insights aligned with clinical guidelines [1]. We
engineered a structured prompting framework using Claude 3.5 Sonnet (Anthropic API) that comprises
three complementary techniques:
1. Few-Shot Exemplars: We provided 5-10 representative examples mapping extracted FHR features
to FIGO classifications and recommended interventions [18]. Each exemplar includes: (i) raw feature vector,
(ii) predicted risk score, (iii) clinical interpretation, and (iv) tiered management steps. This grounds the
model in domain expertise and reduces irrelevant outputs by approximately 40% (pilot evaluation, n=20
cases).
2. Chain-of-Thought (CoT) Prompting: The prompt forces explicit, systematic reasoning through
five sequential steps: (1) Identify FHR pattern (Category I/II/III) based on baseline, variability, and de-
celeration characteristics; (2) Consider predicted acidosis risk (pH 7.15) and model confidence; (3) Link
pattern to pathophysiology (e.g., uteroplacental insufficiency, cord compression); (4) Consult established
FIGO/NICHD guidelines for management thresholds; and (5) Formulate tiered recommendations (monitor,
intrauterine resuscitation, prepare for operative delivery) [18] [19].
3. Retrieval-Augmented Generation (RAG): The LLM was coupled with a searchable knowledge
base of 45 FIGO and NICHD guideline excerpts. During inference, relevant passages are retrieved via
cosine similarity (threshold = 0.75) and injected into the prompt, ensuring recommendations are grounded
in evidence-based practice and reducing hallucinations [20] [19].
The complete prompt structure is documented in Appendix B.1 and illustrated in Figure 4.3.

5.2 Feature Fusion: Synergistic Integration of Hand-Crafted and Learned Rep-
resentations
5.2.1 Motivation and Experimental Design
While hand-crafted features are interpretable, they may miss subtle, non-linear patterns in FHR dynamics
that are predictive of acidosis. Deep learning models (e.g., LSTMs) can automatically learn such patterns
but sacrifice interpretability. A hybrid approach—fusion of hand-crafted and learned features—leverages
the strengths of both: explicit, clinically grounded interpretability combined with data-driven, non-linear
comprehensiveness.

We constructed a three-way comparison: (1) Hand-crafted only (4 Azarnir features), (2) Learned only
(128-dimensional LSTM embeddings), and (3) Fused (concatenation of all 132 dimensions). All models were
trained using logistic regression as a unified classifier to isolate the effect of feature set composition rather
than algorithmic choice.

6 CHAPTER 6: DISCUSSION
6.1 Summary of Proposed Approach
This thesis makes a systems-level argument: the primary barrier to clinically useful fetal monitoring AI is
not a single “best model,” but the absence of scaffolding that connects raw clinical signals to reproducible
evaluation and, ultimately, to safe decision support. The proposed three-tier framework operationalizes that
argument by pairing (i) infrastructure that standardizes cohort extraction and feature computation, (ii)
hybrid modeling that fuses interpretable physiology-derived features with learned representations, and (iii)
evidence-linked narrative generation that translates quantitative predictions into guideline-aligned clinical
language.
In other words, the technical novelty is not only algorithmic; it is the deliberate coupling of repro-
ducibility (Tier 1) with predictive performance (Tier 2) and clinician-facing interpretability
and safety (Tier 3). Chapter 5 quantified each tier’s performance; this chapter explains why those num-
bers matter, what they reveal about hidden bottlenecks (data distribution, false-alarm tolerance, and LLM
failure modes), and what validation pathway is required before bedside use.
6.2 Interpretation of Results and Performance Gaps
The three complementary research threads produced distinct yet interconnected findings that illuminate
both progress and persistent challenges in computational fetal monitoring.
53
6.2.1 Infrastructure Contribution and Reproducibility Impact
The infrastructure result is strategically important because it changes what questions are feasible to ask. A
98% reduction in cohort assembly time (hours →minutes) does more than save labor: it enables independent
benchmarking across multiple models and datasets, which is the only way to surface hidden dataset-specific
failure modes that would otherwise remain “paper-only” performance claims. In that sense, the platform
converts a field-wide bottleneck (70–80% time spent on data plumbing) into capacity for replication and
stress-testing, directly supporting RQ1’s goal of reproducible, shareable cohorts [1] [6].
Reliability metrics matter here because downstream clinical claims are only as trustworthy as upstream
data integrity. Achieving 99.8% feature completeness and a 0.02% error rate (Table 5.15) means
performance comparisons (e.g., AUROC deltas across models) are less likely to be artifacts of missingness,
schema drift, or silent preprocessing differences—precisely the failure modes that undermine reproducibility
in clinical AI [1] [3].
6.2.2 Feature Engineering and Algorithmic Benchmarking
The benchmarking result reframes a common misconception in the literature: high headline performance
does not automatically transfer across institutions. Replicating Azarnir’s OCGP approach on institutional
data produced 78% AUROC versus a reported 96%, illustrating how algorithmic sophistication and cohort
heterogeneity interact—i.e., performance is jointly determined by the model and the data-generating process
[5]. This is exactly the kind of insight that becomes visible only when infrastructure makes independent
replication inexpensive and routine (RQ1 →RQ2 linkage).
The feature-level findings add clinical meaning to the benchmark numbers. Short-term variability (STV)
dominates the anomaly detector, contributing 42.1% of capability (Table 5.2), which is consistent with the
physiological role of autonomic variability as a marker of fetal reserve [8]. More importantly, the fusion result
shows that interpretability and performance are not mutually exclusive: a statistically significant +0.038
AUROC gain (p = 0.023) corresponds to 6 additional correctly classified cases per 100 deliveries while
preserving feature-level auditability (Section 5.2; Figure 4.2) [2]. This supports RQ2’s central claim that
hybrid representations are a practical path to clinically acceptable accuracy without sacrificing transparency.
6.2.3 LLM Clinical Translation and Safety
The LLM finding is best interpreted as a bottleneck diagnosis rather than a celebration of a single metric.
Achieving = 0.82 indicates that structured prompting can turn free-form language generation into a
reproducible clinical artifact that meets FDA guidance for decision-support agreement ( ≥0.80) [1] [4]. The
ablation results make the mechanism explicit: unstructured prompting yields moderate agreement ( = 0.45),
while the fused prompting stack (few-shot + CoT + RAG) is the only configuration that reliably crosses
the regulatory threshold (Table 5.17). This implies that prompt structure and grounding—not model size
alone—are critical levers for safe deployment.
At the same time, the evaluation design constrains what can be claimed. The 100-case, 2-rater retro-
spective study is appropriately framed as proof-of-concept; it demonstrates feasibility and identifies failure
modes, but it cannot bound rare safety events or generalize across practice environments. Moving from
“regulatory-aligned” to “regulatory-ready” therefore requires multi-center sampling and more independent
raters (3–5) at larger scale (500–1,000+), which this thesis explicitly treats as the next validation milestone
[1].
Structured prompting substantially improved safety: 0% harmful recommendations versus 8% base-
line, representing a 100% reduction in unsafe suggestions (Table 5.17). The ablation study confirmed
that no single prompting component alone achieves 0.80; synergy between few-shot exemplars, chain-of-
thought reasoning, and retrieval-augmented generation is required (Table 5.17).
6.3 Clinical Impact and Strategic Value
6.3.1 Objectivity and Consistency
The framework addresses the well-documented inter-observer variability in visual CTG interpretation (Co-
hen’s = 0.60-0.75) by providing quantitative risk scores and automatically highlighted segments. By stan-
54
dardizing feature extraction and guideline adherence through structured prompting, the system reduces
subjective interpretation that contributes to both delayed interventions and unnecessary procedures [1] [6].
6.3.2 Timeliness of Decision-Making
The continuous Health Confidence Score (HCS) shifts decision-making from binary labeling toward trend
interpretation, which is closer to how clinicians reason during labor (progressive deterioration versus in-
stantaneous “normal/abnormal”). This directly connects to the motivating question in Chapter 1—when
to intervene and with what confidence—because the value of a score is not only its threshold crossing, but
its trajectory and rate of change [3]. In parallel, flagging high-attention 60-second windows operationalizes
“what to look at next”: it narrows clinician review to the most influential segments, which can reduce the
cognitive burden of scanning long tracings during time-critical scenarios [5].
6.3.3 Educational and Quality Assurance Value
The LLM-generated narratives, grounded in FIGO and NICHD guidelines, serve as real-time educational
tools for trainees while providing audit trails for quality assurance. The transparency of hand-crafted features
combined with attention-weight visualization creates a ”first-order explanation” mechanism that supports
clinical trust and regulatory audit requirements [1].
6.4 Limitations and Critical Challenges
6.4.1 Data Quality and Generalizability
Despite infrastructure improvements, FHR signals remain inherently noisy, with artifacts from maternal
movement, ultrasound probe displacement, and signal loss affecting up to 30% of recordings [1]. The
single-institution dataset, while substantial, reflects specific patient demographics and clinical protocols at
Stony Brook University, limiting generalizability to other maternity units with different baseline risk profiles
[1] [7].
6.4.2 Model Transparency and Causal Insight
While attention maps offer insight into which segments influenced predictions, they do not provide causal
explanations linking specific patterns to pathophysiology (e.g., uteroplacental insufficiency). The LLM’s
reasoning process, even with chain-of-thought prompting, remains a statistical association rather than a
mechanistic model of fetal physiology. Deeper causal inference methods are needed to establish clinical trust
beyond correlation-based predictions.
6.4.3 LLM Reliability and Hallucination Risk
Even with grounding, LLM failure modes do not disappear—they change shape. In our evaluation, RAG
reduces harmful recommendations relative to baseline prompting (8% →2%), and the fused prompting
stack eliminates them in-sample (0%) (Table 5.17). However, residual risks remain: the model can still
produce plausible-sounding but incorrect guideline citations or overconfident language in edge cases that
are underrepresented in exemplars. This motivates two deployment requirements: external auditing (to
characterize error modes across sites) and dynamic safety monitoring (to detect distribution shift and citation
failures) as recommended by safe-LLM clinical frameworks [1].
6.4.4 Ethical and Regulatory Hurdles
Clear accountability frameworks remain unresolved: Who is liable when an LLM-generated recommendation
contributes to adverse outcomes? Privacy protection under HIPAA requires that prompts contain no patient
identifiers, yet clinical context often demands specific details that could enable re-identification [1]. Alignment
with FDA Software as a Medical Device (SaMD) regulations demands prospective validation showing no
increase in adverse outcomes compared to standard care [1] [4], a study this thesis does not provide.
55
6.4.5 Evaluation Scope and Prospective Validation
All experiments remain retrospective, lacking prospective studies demonstrating impact on intervention rates
and neonatal outcomes [1] [7]. The LLM evaluation involved only 100 cases reviewed by two obstetricians;
multi-center studies with 500–1,000 cases are required for regulatory submission [1]. Prospective randomized
controlled trials (RCTs) are necessary to prove that AI-assisted interpretation improves outcomes without
increasing unnecessary operative deliveries. While κ= 0.82 meets FDA guidance, the 95% CI [0.73–0.90] and
evaluation on 100 retrospective cases by 2 raters reflect proof-of-concept validation; prospective multi-center
studies with 500–1,000 cases and 3–5 raters are required for regulatory submission [21].
Milestone-driven clinical validation roadmap (timeline + sample sizes). To move from retrospec-
tive performance to regulatory-grade evidence, we propose a phased pathway with explicit time horizons and
cohort sizes:
• Phase 1 (Months 0–12): Multi-center prospective cohort validation (n=500). Prospec-
tive observational study across multiple sites (e.g., 5 tertiary centers; ≈100 consecutive intrapartum
cases/site). Primary endpoint: agreement between system predictions and umbilical artery pH (tar-
get κ > 0.75); secondary endpoints: FIGO guideline adherence, false-alarm rate, and usability (e.g.,

7.3.1 Limitation of Pilot-Scale Validation (n=100, 2 Raters)
The LLM safety evaluation constitutes a Phase 0 pilot study (n=100 retrospective cases, 2 expert raters)
designed to establish feasibility and coarse statistical bounds. While this scope is sufficient for a computa-
tional proof-of-concept, FDA-grade validation requires prospective, multi-center studies with substantially
larger cohorts (n=500–1,000) and 3–5 independent clinicians (see Section 6.4.5) [21].
Institutional Bias Quantification: The single-institution cohort (Stony Brook University Hospital)
reflects a higher-acuity tertiary-care risk mix, characterized by a 35% cesarean delivery rate and 30%
fetal acidosis prevalence (Table 4.1). The cesarean rate is modestly higher than the U.S. national baseline
(≈32%) reported in national vital statistics [24]. This higher baseline-risk profile may limit generalization
to lower-acuity community settings, potentially shifting operating points and false-alarm burden; therefore,
external validation (CTU-UHB) and planned multi-center prospective validation are required to substantiate
generalizability [5].

A.1 Preprocessing (4 Hz, 10-minute window)
# Input:
# fhr_bpm: array length N=2400 (4 Hz, 10-minute window), may contain NaN
# toco: array length N=2400 (optional, for deceleration timing)
# Output:
# fhr_clean: cleaned array with short gaps interpolated
# valid_mask: boolean mask for samples usable in variability metrics
def preprocess_fhr(fhr_bpm):
# 1) Identify missing samples
mask = ~np.isnan(fhr_bpm)
# 2) Fill short gaps (<15s => <60 samples) by linear interpolation
f = fhr_bpm.copy()
gap_runs = find_nan_runs(f) # returns list of (start,end) inclusive
for s,e in gap_runs:
gap_len = e - s + 1
if gap_len < 60 and s > 0 and e < len(f)-1:
f[s:e+1] = np.linspace(f[s-1], f[e+1], gap_len+2)[1:-1]
# 3) Mark long gaps as invalid for STV/variability calculations
valid = ~np.isnan(f)
for s,e in gap_runs:
if (e - s + 1) >= 60:
valid[s:e+1] = False
return f, valid
A.2 Baseline FHR (bFHR) — iterative trimming
Note: This pseudocode implements the baseline extraction logic described by Azarnir et al. [6], translated
into executable-style Python/NumPy form for reproducibility.
def compute_baseline_fhr(fhr_clean, eps=0.5):
# Iterative trimming of deviations >15 bpm from current median
x = fhr_clean[~np.isnan(fhr_clean)]
b_prev = np.median(x)
while True:
keep = np.abs(x - b_prev) <= 15
if np.all(keep):
return round(b_prev, 1)
x2 = x[keep]
b_new = np.median(x2)
if abs(b_new - b_prev) < eps:
return round(b_new, 1)
x = x2
b_prev = b_new

A.3 Short-Term Variability (STV) — RMSSD in ms
Note: This pseudocode implements the STV/RMSSD definition described by Azarnir et al. [6], translated
into executable-style Python/NumPy form for reproducibility.
def compute_stv_rmssd_ms(fhr_clean, valid_mask):
# Convert bpm -> RR interval in seconds
f = fhr_clean.copy()
f[~valid_mask] = np.nan
rr = 60.0 / f
rr = rr[~np.isnan(rr)]
if len(rr) < 2:
return np.nan
diffs = np.diff(rr)
rmssd_s = np.sqrt(np.mean(diffs**2))
return round(rmssd_s * 1000.0, 2) # milliseconds
A.4 Acceleration Count (AC) — NICHD criteria
def count_accelerations(fhr_clean, bFHR):
# Acceleration: >=15 bpm above baseline for >=15 seconds (>=60 samples at 4 Hz)
above = fhr_clean >= (bFHR + 15)
runs = find_true_runs(above) # list of (start,end)
accel = 0
for s,e in runs:
if (e - s + 1) >= 60:
accel += 1
return accel

def classify_decelerations(fhr_clean, toco, bFHR):
# Detect decelerations: <=15 bpm below baseline for >=15 seconds (>=60 samples)
below = fhr_clean <= (bFHR - 15)

counts = {"early":0, "late":0, "variable":0, "prolonged":0}
